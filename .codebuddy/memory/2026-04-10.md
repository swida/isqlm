# 2026-04-10

## Fixed: Async query error not reported (shows "Query OK, -1 rows affected" instead of error)

**Problem**: When a SELECT query fails with a MySQL error (e.g. ERROR 1690 BIGINT out of range), the async path in isqlm shows "Query OK, -1 rows affected" instead of the actual error message.

**Root cause**: `mysql_real_query_nonblocking` returns `NET_ASYNC_COMPLETE` even when the query fails with a server error. The C code (`mysql-query-start`, `mysql-query-continue`) did not check `mysql_errno(conn)` after `NET_ASYNC_COMPLETE`, so the error was silently swallowed. Subsequently, `mysql_store_result_nonblocking` returned NULL (no result set), and `isqlm--async-process-result` fell through to the DML branch which showed `mysql_affected_rows` = -1.

**Fix (3 layers)**:
1. **C layer (`mysql-el.c`)**: Added `mysql_errno(conn)` check after `NET_ASYNC_COMPLETE` in `Fmysql_query_start` and `Fmysql_query_continue` — signals error to Emacs instead of returning `complete`.
2. **C layer (`mysql-el.c`)**: Added `mysql_errno(conn)` check in `Fmysql_store_result_start` when `res == NULL && field_count != 0` — second safety net.
3. **Elisp layer (`isqlm.el`)**: Added defensive branch in `isqlm--async-process-result` for SELECT queries with nil `res-ptr`. Also added `(>= affected 0)` guard in both sync and async DML branches to prevent displaying negative affected-rows.

## Fixed: Error messages now use mysql-client format

**Problem**: isqlm displayed errors as "*** Error *** Query failed (no result set)" or "*** Error *** mysql_real_query_nonblocking: BIGINT value is out of range..." instead of the mysql client format "ERROR 1690 (22003): BIGINT value is out of range in '(...)'"

**Fix**:
1. **C layer (`mysql-el.c`)**: Added `mysql-errno`, `mysql-error`, and `mysql-sqlstate` functions to expose MySQL error state to Elisp.
2. **Elisp layer (`isqlm.el`)**: Added `isqlm--format-mysql-error` to format errors as "ERROR <errno> (<sqlstate>): <message>\n" (mysql client style). Added `isqlm--output-mysql-error` helper that prioritizes MySQL error format, falling back to condition-case message.
3. Replaced all SQL-related `*** Error ***` outputs with `isqlm--output-mysql-error` calls across async, sync, script, and quick-sql paths.

## Fixed: "*** Error *** Unknown error" shown instead of MySQL error (2026-04-11)

**Problem**: Async SELECT queries that fail (e.g. ERROR 1690 BIGINT out of range) showed "*** Error *** Unknown error" instead of the actual MySQL error.

**Root cause**: After the async query completes, `mysql_errno(conn)` is correctly non-zero in the query phase. But by the time `isqlm--format-mysql-error` is called from Elisp (often after store-result), `mysql_errno(conn)` may have been reset to 0. The `signal_mysql_error` C function was formatting messages as `"prefix: message"` (e.g. `"mysql_real_query_nonblocking: BIGINT..."`) which didn't match MySQL CLI format. And callers of `(isqlm--output-mysql-error)` without the `err` argument hit the "Unknown error" fallback.

**Fix (2 layers)**:
1. **C layer (`mysql-el.c`)**: Changed `signal_mysql_error` to emit MySQL CLI format: `"ERROR <errno> (<sqlstate>): <message>"` instead of `"prefix: message"`. The `prefix` parameter is now unused (kept for API compat, marked with `(void)`).
2. **C layer (`mysql-el.c`)**: In `Fmysql_store_result_start`, when `res == NULL && field_count != 0`, signal error even if `mysql_errno == 0` (fallback: "No result set returned for SELECT").
3. **Elisp layer (`isqlm.el`)**: `isqlm--output-mysql-error` now checks if the `err` message already matches MySQL CLI format (`"ERROR \d+ (\w+): ..."`) and outputs it directly instead of wrapping with `"*** Error ***"`.
